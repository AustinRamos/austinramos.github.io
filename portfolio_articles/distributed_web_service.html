What's it all about?

Ever heard of an RPC? Well this is basically that, but worse. Using Sun RPC would have probably made this project just slightly more than trivial, but if I had jumped straight into RPC's i probably would not understand very much about what goes on in even the simplest concurrent server/client communication. they basically generate code to allow you to (seemingly) execute a procedure from one process on another process with a different memory space . For example, by simply calling a simply function like add_tuple() on the client side, the RPC would act as a black box, executing that functionality on the server side as if it had been on executed on the client side. It basically hides all the complexity of establishing connections and telling the server what kind of actions the client expects it to perform. Its a clever trick and vastly reduces the complexity of how the client and server interact. Its pretty much the concept i used to develop the static library I described below.
I mean, look how simple a client program seems:
int main(void) {
int y = num_items(qname);
printf("Number of items: %d \n", y);
float fl = 2.3;
set_value( 1,"test1",fl ,qname);
set_value( 2,"test2",fl ,qname);
set_value( 1,"test1",fl ,qname);
y = num_items(qname);
delete_key(1, qname);
}
Now if you want a headache to look at the libkeys source file( keys.c). Thats where the messy code dealing with establishing a connection to the server as well as sending and receiving messages. It was SO annoying to write. But now the "clients" can write little programs using an api that executes commands remotely(on the server side) designed from the perspective of the client( with all the complexity hidden).

Static what?


The "Invoke service" and "Request response" portions of the diagram above represent the role of the static library in this Distributed Service. it is compiled and linked with the client code(as per the makefile), so that, rather than the client side programmer does not have to concern him/herself with establishing a connection, and making the the server side knows what it is doing. I say this because it is not just possible to send a single message to the server and have it know if it is adding a tuple, removing, modifying, etc.
The client has to send a message before hand telling the server what to expect, so the server knows what to do with the message containing the real data, and also so that it knows what it is supposed to return to the server. I decided to do this in a particular way. First I sent a "dummy" message struct, where the key value integer is just an integer from 1-6. these represent the 6 possible operations the server side will allow for the client. For some, such as num_items() and init(), there is now need to send 2 messages. However for others like set_value() and delete_key() it is necessary to first tell the server what it is supposed to do, and the send the corresponding data(for set_value() a full struct to be added, and for delete_key another "dummy" struct where the key value is the struct whose key that is to be deleted.

Generating Executables

I have found, at least on my virtual machine(linux mint) that the message queues are very finicky. There is a default msg_max size less than the size of my request struct, so it is important to run the following shell command “ sudo sh -c 'echo 512 > /proc/sys/fs/mqueue/msg_max' “.
I was also able to solve many issues by deleting the leftover empty queues from the disk(as they are stored on the actual linux os). They can be found in the directory “/dev/mqueue” and can be removed with a simple “rm” command. To build the two executables, all that is necessary is to run the command “make” in the linux terminal. This will compile the server executable, compile the keys.c code, archive it into a static library, and handle all the static library and pthread linking.
The demos would then be available by running “./server” on one terminal, and “./client#” on another. There are two demo client1, client2 , which show different functionalities Client1.c will check the the number of items is correctly 0 on start. It will add 1 tuple, show that the method on the server side returned that it was a success. Then it will check the new total number of elements stored, which will be 1. It then adds another tuple. Shows totalnum =2; It deletes 1, shows total num = 1; Then it will init() And show server side has been reset and totalnum is at 0 again. Client2.c- Will add 2 tuples. Will get 1 tuple and print it out. Will change a tuple and print it out.
