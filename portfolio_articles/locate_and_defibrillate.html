Locate and Defibrillate

This was a really fun final project I worked on for a class titled Software Engineering Workshop. The goal was to step through the design process for a Minimum Viable Product(MVP), treating it as though it were to become a fully fledged consumer product. All around, this was a much less technical process than I am used to. We proposed an IOS application that would be able to help students find the closest defibrillator to them. This was designed for situations where these students found themselves bystanders in cardiac emergencies.
It was cool to step through the process, I was lucky enough to have Product Managers at local startups to help guide us through the process of setting deadlines and helping us through different design decisions(tradeoffs of using a non native ios development platform like react, Xamarin, or Cordoba, database schemas on the backend, UI design decisions, etc..)

Check out the full project on my Github!

What does it do(and how)?

This specific project was designed for students on our campus to be able to find the location of the nearest defibrillator in the event of a cardiac emergency using a native iPhone application. There are two components the front end and the back end. The front end is of course, the iPhone application, while the back end is a very simple administrator site, database, and collection of defibrillator locations, all stored on an AWS ec2 instance (Amazon web server).

Architecture:


When the student opens the app on their side, it will present a cache of the locations it had received from the server last time, while it attempts to make a request to the server to get the most up to date locations.



They are presented with a map showing their location and all the pins representing different location. They can press any pin directly to get the building name and brief description of its location in the building. When they press the giant red locate AED button, the application will isolate the shortest path.
On the back end, there is a relatively crude and simple administrator site, a database holding the (properly encrypted) credentials of the authorized administrators and a simple csv text file holding the locations of the defibrillators in a way that is very simple to parse on the front end side of things(This could have been designed in a more robust manner, but given the circumstances we opted for simplicity.). Authorized administrators can log on and see the list as well as add, modify, or remove locations.
Uhh, that’s pretty much it! If we had the goal of making a real end product, we would have changed or improved certain aspects of our design(I discuss this below in the user testing and feedback section), but this was just a fun project where for the first time I got to really think about different design choices and their impact on both the rest of the development process as well as the user experience.
The cool thing is Defibrillators aside, this MVP would work for any number of on campus situations where a it would be helpful for a group of students to know the exact location of a number of different particular objects or buildings, so long as there is a group of trusted administrators to update and maintain the locations.

Of course, given the 'MVP' nature of this project, there are naturally a number of things we could do to refine it if it were to be used in actual emergency situations. Here are the most significant things we would have changed or improved on if this were to be a proper 'product':

  • A much cleaner backend
  • More secure way of storing locations
  • Better caching mechanism
  • More sophistaced shortest path finding(we are using Euclidian distance, perhaps not the best for all campuses but ours is mostly traverable in this manner).
  • Giving directions to the actual defibrillator rather than just isolating it from the others.
  • A feature to contact authorities/paramedics

User Testing and Feedback

We had all the students on the Emergency Support Team, as well as Residential advisors and other student advisors test the app, and got great feedback! The PM’s who were guiding us through the project constantly were bringing up the importance of user testing and user feedback, something I have never actively sought in previous projects. Below is the information regarding the user testing:
We had 67 people use the application and give feedback:

  • EST members (35), other students (32), RAs/WUSAs/faculty (7/67)
  • EST members (including RAs and WUSAs) were ones to actually use the app

Survey prompts after they used the application included:
  • Would you download the application
  • how likely would you be to recommend it to a friend?
  • would you be disappointed without its existence on the market?
  • any comments of app importance/features (free response)
